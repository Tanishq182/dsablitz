## Q&A Set 1: Gin Router Path Parameter Binding

### 1. Interviewer Intent
The interviewer wants to check the candidate's familiarity with Go web routing frameworks (like Gin), how request routes are structured, and how to retrieve path variables (like lobby codes) securely.

### 2. Strong Answer
In Go backend development, **Gin** is a high-performance HTTP web framework that utilizes a custom Radix tree router. It allows defining dynamic route patterns containing wildcards or path parameters.

To bind dynamic variables in Gin, we specify parameters using colon syntax in our route definitions (e.g. `/rooms/:code`). Inside the HTTP handler, we call `c.Param("code")` to extract the parameter value from the request context.

For JSON payloads, we define a struct with field tags mapping to JSON keys (e.g. ``json:"ready"``) and call `c.ShouldBindJSON(&req)` to deserialize the request body directly into our Go struct, validating it before passing the data to the service layer.

### 3. Common Mistakes
* Manually parsing the URL string using standard string splits instead of using the framework's native parameter binding features.
* Accessing path parameters without checking if they are empty, which can cause routing issues.
* Forgetting to handle JSON binding errors, allowing requests with malformed JSON payloads to proceed to the service layer.

### 4. Follow-up Questions
* **How does Gin handle requests targeting routes that do not exist?**
  * *Answer*: Gin returns a default `404 Not Found` response. We can customize this by overriding the router's `NoRoute` handler to return a structured JSON response.

## Q&A Set 2: UUID Primary Keys vs Auto-Incrementing Integers

### 1. Interviewer Intent
The interviewer wants to assess the candidate's understanding of database security, ID scanning vulnerabilities, and the tradeoffs of using random UUID primary keys compared to traditional auto-incrementing integers.

### 2. Strong Answer
Relational databases traditionally use auto-incrementing 64-bit integer IDs (`BIGSERIAL`) for primary keys. However, sequential IDs are vulnerable to ID scanning attacks, where an attacker guesses database record volumes and accesses other users' data by incrementing the ID in the API URL (e.g. `/api/battles/42`).

We resolve this by using **UUIDs** (specifically random UUIDv4) generated by PostgreSQL's `gen_random_uuid()` function. UUIDs are 128-bit cryptographically random values, making it impossible to guess active IDs. UUIDs also allow client-side ID generation without collisions, simplifying offline workflows.

However, UUIDs have tradeoffs: they take 16 bytes of storage instead of 8 bytes for integers, and because random UUIDs are inserted out of order, they cause B-tree index fragmentation, which can degrade database write performance over time.

### 3. Common Mistakes
* Assuming UUIDs are completely free of performance tradeoffs (ignoring index fragmentation and storage overhead).
* Using sequential integers in public URLs while claiming the API is secure.
* Generating UUIDs in the application layer without checking for nil values before database insertion.

### 4. Follow-up Questions
* **How can we mitigate B-tree index fragmentation caused by random UUIDs?**
  * *Answer*: Partition large tables (like `submissions`) by date range to isolate write workloads, and run vacuum schedules regularly to optimize index sizes.

## Key Takeaways
1. **Gin routers** parse dynamic path parameters using colon syntax and provide native binding functions.
2. **UUID primary keys** prevent ID scanning attacks but increase index fragmentation and storage overhead compared to integers.
3. **JSON binding validation** protects the service layer from malformed payloads.

---

## Interview Questions
* **What is index fragmentation in a relational database?**
  * *Answer*: It occurs when random inserts force index pages to split, leaving empty space in the index pages and slowing down index scans.
* **Why does Gin use a Radix tree structure for routing?**
  * *Answer*: The Radix tree minimizes memory footprint and provides fast path lookup, making routing time independent of the number of registered routes.

---

## Common Mistakes
* **Sequential integer leak**: Exposing auto-incrementing database IDs in public API endpoints, exposing the system to enumeration attacks.
* **Skipping binding errors**: Omitting error checks during JSON deserialization, causing Nil pointer dereference bugs in the handler.
